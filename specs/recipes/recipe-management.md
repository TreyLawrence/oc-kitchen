# Feature: Recipe Management

## Overview

Recipes are the core entity in OC-Kitchen. They come from three sources: imported from the web (Bon Appetit, NYT Cooking, Woks of Life, and others), generated by Claude based on user preferences, or entered manually. The system tracks which recipes were hits and which fell flat, building a preference profile over time that informs future suggestions.

## User Stories

### Creating Recipes
- As a user, I can import a recipe from a URL and have it parsed into structured data (title, ingredients, instructions, etc.)
- As a user, I can ask the agent to "find me something new" and it browses my favorite food sites for recipes to suggest
- As a user, I can ask Claude to generate a recipe based on a prompt (e.g., "something smoky on the Big Green Egg for 4 people")
- As a user, I can create a recipe manually by entering all fields
- As a user, I can edit any recipe after creation regardless of its source
- As a user, I set my favorite recipe sources (food blogs/sites) during onboarding, and the agent draws from them when discovering new recipes

### Browsing & Organizing
- As a user, I can browse all my recipes with filters (by source, verdict, tags, favorites)
- As a user, I can search recipes by title, ingredient, or tag
- As a user, I can tag recipes with custom labels (e.g., "weeknight", "date night", "meal prep")
- As a user, I can mark a recipe as a favorite for quick access

### Rating & Feedback
- As a user, after cooking a recipe I rate it on a four-tier scale:
  - **Banger** — all-time favorite, heavy rotation in meal plans
  - **Make again** — solid recipe, regular rotation
  - **Try again with tweaks** — has potential but needs changes, resurfaces less frequently with my notes visible
  - **Don't make again** — hard blocked from future suggestions
- As a user, I can add personal notes to a cook log entry (e.g., "needed more salt", "sub fish sauce for soy")
- As a user, I can see a history of when I cooked a recipe and what I rated it each time

### Preference Learning
- As a user, the system learns my preferences from my ratings, notes, and conversation history
- As a user, I get better recipe suggestions over time based on what I've liked and disliked
- As a user, recipes are suggested that match my kitchen equipment (Big Green Egg, Instant Pot, etc.)

## Data Model

See `shared/data-model.md` for the full schema. Key tables:

- `recipes` — core recipe data, source tracking, rating, favorites
- `recipe_ingredients` — structured ingredient list with categories
- `recipe_cook_log` — **NEW**: tracks each time a recipe is cooked, with rating and notes per cook
- `user_preferences` — **NEW**: stores learned preferences per user (cuisine affinities, ingredient preferences, equipment, dietary constraints)
- `user_equipment` — **NEW**: kitchen equipment inventory per user

## Tool Contracts

These are OpenClaw plugin tools registered via `api.registerTool()`. The agent calls them based on conversation context and skill instructions.

### `create_recipe`
Create a recipe manually.

**Parameters:**
```json
{
  "title": "Smoked Pork Shoulder",       // required
  "description": "Low and slow on the Big Green Egg",
  "source": "manual",                     // required: "manual" | "imported" | "ai_generated"
  "servings": 8,
  "prepMinutes": 30,
  "cookMinutes": 480,
  "instructions": "## Steps\n1. Rub the shoulder...",  // required
  "ingredients": [
    { "name": "pork shoulder", "quantity": 8, "unit": "lbs", "category": "protein" },
    { "name": "yellow mustard", "quantity": 0.5, "unit": "cup", "category": "pantry" }
  ],
  "tags": ["bbq", "big green egg", "low and slow"]
}
```

**Success:** `{ "ok": true, "recipe": { "id": "abc123", "title": "...", ... } }`

**Error:** `{ "ok": false, "error": "Title is required" }`

### `import_recipe`
Import a recipe from a URL. Uses JSON-LD schema.org/Recipe extraction first, falls back to LLM parsing.

**Parameters:**
```json
{
  "url": "https://www.bonappetit.com/recipe/slow-roast-gochujang-chicken"
}
```

**Success:** `{ "ok": true, "recipe": { ... }, "parseMethod": "json-ld" | "llm" }`

**Error:** `{ "ok": false, "error": "Could not parse recipe from this URL" }`

### `discover_recipes`
Browse the user's favorite recipe sources to find new recipes matching a query or theme. Uses the `favorite_sources` from user preferences.

**Parameters:**
```json
{
  "query": "weeknight chicken",        // optional search theme
  "cuisine": "korean",                 // optional cuisine filter
  "sources": ["bonappetit.com"],       // optional override; defaults to user's favorite_sources
  "count": 3                           // how many suggestions (default 3)
}
```

**Process:**
1. Fetches the user's `favorite_sources` from preferences (unless overridden)
2. Uses web search scoped to those sites to find matching recipes
3. Returns summaries (title, URL, brief description) — does NOT auto-import
4. User picks which ones to import via `import_recipe`

**Success:**
```json
{
  "ok": true,
  "suggestions": [
    {
      "title": "Gochujang Chicken",
      "url": "https://www.bonappetit.com/recipe/slow-roast-gochujang-chicken",
      "source": "bonappetit.com",
      "description": "Spicy-sweet roasted chicken thighs with gochujang glaze"
    },
    {
      "title": "Korean Fried Chicken",
      "url": "https://cooking.nytimes.com/recipes/korean-fried-chicken",
      "source": "cooking.nytimes.com",
      "description": "Extra-crispy double-fried chicken with a soy-garlic glaze"
    }
  ]
}
```

### `generate_recipe`
Ask Claude to create a recipe based on a prompt, factoring in user preferences and equipment.

**Parameters:**
```json
{
  "prompt": "A weeknight stir fry using the wok, something with crispy tofu",
  "maxMinutes": 45,
  "servings": 2,
  "equipment": ["wok"]
}
```

**Success:** `{ "ok": true, "recipe": { ... } }` (automatically saved with `source: "ai_generated"`)

### `search_recipes`
Search saved recipes with optional filters.

**Parameters:**
```json
{
  "query": "chicken",            // free-text search across title, ingredients, tags
  "source": "imported",          // "manual" | "imported" | "ai_generated"
  "verdict": "banger",           // "banger" | "make_again" | "try_again_with_tweaks"
  "favorite": true,
  "tags": ["weeknight"],
  "limit": 20,
  "offset": 0
}
```

**Success:** `{ "ok": true, "recipes": [...], "total": 47 }`

### `get_recipe`
Get a single recipe with ingredients and full cook history.

**Parameters:** `{ "id": "abc123" }`

**Success:**
```json
{
  "ok": true,
  "recipe": {
    "id": "abc123",
    "title": "...",
    "ingredients": [...],
    "cookLog": [
      {
        "id": "log1",
        "cookedAt": "2026-04-20T...",
        "verdict": "banger",
        "notes": "Great but needed more acid",
        "modifications": [
          { "original": "4 cloves garlic", "modification": "8 cloves garlic" }
        ],
        "photos": ["/path/to/photo1.jpg"]
      }
    ]
  }
}
```

### `update_recipe`
Update recipe fields. Partial updates — only include fields to change.

**Parameters:** `{ "id": "abc123", "title": "New Title", "tags": ["updated"] }`

### `delete_recipe`
Delete a recipe. Sets `meal_plan_entries.recipeId` to null for any plans referencing it.

**Parameters:** `{ "id": "abc123" }`

### `log_cook`
Log a cooking session with verdict, notes, modifications, and photos.

**Parameters:**
```json
{
  "recipeId": "abc123",
  "verdict": "banger",           // required: "banger" | "make_again" | "try_again_with_tweaks" | "dont_make_again"
  "notes": "Served with rice from the Zojirushi. Incredible.",
  "modifications": [
    { "original": "4 cloves garlic", "modification": "8 cloves garlic" },
    { "original": "1 tsp chili flakes", "modification": "1 tbsp gochugaru" }
  ],
  "photos": ["/path/to/photo.jpg"]
}
```

**Success:** Returns the cook log entry. Also updates the recipe's `verdict` field to match.

## Behavior Rules

1. **Title is required.** Instructions are required. All other fields are optional.
2. **Rating is a four-tier system:** `banger`, `make_again`, `try_again_with_tweaks`, `dont_make_again`. No numeric stars.
3. **Recipe-level verdict** is derived from the most recent cook log entry's rating. A recipe can change tiers over time (e.g., "try again with tweaks" → "banger" after modifications work out).
4. **Source is immutable** after creation — a manual recipe stays manual, an imported recipe stays imported.
5. **Tags** are stored as a JSON array. There are three kinds:
   - **Equipment tags** — auto-derived from recipe content (e.g., "big green egg", "instant pot", "wok"). Matched against the user's equipment list.
   - **Duration tags** — auto-derived from total time (prep + cook): "quick" (< 30 min), "weeknight" (< 60 min), "project" (2+ hours). No tag for 60-120 min.
   - **User tags** — free-form, manually added by the user.
6. **Deleting a recipe** does not cascade to meal plan entries — it sets their `recipeId` to null and preserves `customTitle` as a fallback.
7. **Importing** must handle at minimum: Bon Appetit, NYT Cooking, Woks of Life. Other sites are best-effort.
8. **AI generation** should factor in user preferences (from `user_preferences`) and equipment (from `user_equipment`) when available.
9. **Duplicate detection:** warn (don't block) when importing a URL that already exists in the database.
10. **Cook log** is append-only — you can't edit or delete past cook entries.

## Edge Cases

- Importing a URL that returns 404 or is not a recipe page → error with descriptive message
- Importing a URL from an unsupported site → best-effort LLM parse, warn user that fields may be incomplete
- AI generation when `ANTHROPIC_API_KEY` is not configured → error with setup instructions
- Logging a verdict for a recipe that has never been cooked → not allowed, must log a cook first
- Searching with no filters → returns all recipes, newest first, limited to 20

## Design Decisions

1. **Equipment** — user-profile data collected during onboarding, not hardcoded. See `specs/shared/onboarding.md` (TODO).
2. **Recipe import** — hybrid approach: extract JSON-LD schema.org/Recipe structured data first (free, reliable), fall back to LLM parsing when missing.
3. **Rating system** — four-tier verdict (banger / make again / try again with tweaks / don't make again). No numeric stars. "Don't make again" is a hard block from future suggestions.
4. **Auto-tags** — equipment tags (matched from recipe content against user's equipment list) and duration tags (quick / weeknight / project). All other tags are manual.
5. **Cook log detail** — verdict + free-text notes + structured modifications (original/modification pairs) + photos. No "who cooked" for v1 (single user per account), design for multi-cook families later. Photo storage TBD (local filesystem for dev, S3/R2 for prod).
6. **Preference learning** — dual approach. Lightweight preference summary in the DB (updated periodically from cook history — cuisine affinities, ingredient preferences, common modifications, equipment comfort). Raw recent cook log entries also passed as context to Claude at generation time. DB summary handles big picture; raw context gives Claude the nuance.
